# 進貨作業總覽

## 概述

進貨作業包含五個 API 接口，用於處理採購單的收貨流程，從取得待收貨清單、輸入收貨資料、核對明細到完成過帳。

## 流程圖

```
┌─────────────────────────────────────────────────────────────────┐
│                        R1 取得預期收貨清單                           │
│  (狀態: 未開始) → 取得待收貨的採購單清單                              │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        R2 取得收貨明細資料                           │
│  選定採購單 → 取得 PO 項次、貨號、品名、到期日等明細                     │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        R3 回傳收貨資料                              │
│  逐項輸入收貨資料 (件數、到期日) → 暫存/更新收貨明細                      │
│  (requestFnName: "3.2確認收貨" 暫存, "3.3修改" 更新)                  │
│  (狀態變更: 收貨中)                                                   │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        R4 取得收貨核對明細                            │
│  完成所有項目後 → 取得核對資料 (已收件數、儲位選單、總需求公斤數)              │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        R5 回傳全收確認                              │
│  輸入板數與儲位 → 確認過帳並產生板標籤                                │
│  (狀態變更: 收貨完成)                                               │
└─────────────────────────────────────────────────────────────────┘
```

## 接口列表

| 代號 | 接口名稱 | URL |說明 |
|------|---------|-----|------|
| R1 | 取得預期收貨清單 | POST `/wmService/v1/PO/PoHeaderData` | 取得未完成收貨的採購單清單 |
| R2 | 取得收貨明細資料 | POST `/wmService/v1/PO/PoDetailData` | 依採購單取得收貨明細 |
| R3 | 回傳收貨資料 | POST `/wmService/v1/PO/PoReceivingItem` | 新增/更新收貨資料 |
| R4 | 取得收貨核對明細 | POST `/wmService/v1/PO/PoVerifying` | 取得收貨核對資料與儲位選單 |
| R5 | 回傳全收確認 | POST `/wmService/v1/PO/PoCfmReceipt` | 確認過帳並產生板標籤 |

## 資料傳遞關鍵欄位

流程中需要持續傳遞的關鍵資料欄位：

| 欄位名稱 | 說明 | 來源/用途 |
|---------|------|---------|
| `wmsAsnNumber` | WMS 預期收貨單號 | R1 回傳，後續 R2~R5 必須傳遞 |
| `storerCode` | 貨主代碼 | R1 回傳，後續 R2~R5 必須傳遞 |
| `externReceiptNumber` | 外部收貨編號 (採購單號) | R1 回傳，後續 R2~R5 必須傳遝 |
| `lineNo` | WMS 項次 | R2 回傳，R3 必須傳遞 |
| `recLocCode` | 儲位編號 | R4 回傳，R5 必須傳遞 |

## 單據狀態變化

| 狀態 | 說明 | 可操作的接口 |
|------|------|------------|
| 未開始 | 採購單建立，尚未開始收貨 | R1, R2, R3 |
| 收貨中 | 已開始輸入收貨資料，尚未完成 | R2, R3, R4, R5 |
| 收貨完成 | 全收確認過帳完成 | 僅可查詢 |

## 詳細接口說明

### R1 - 取得預期收貨清單

取得未完成收貨的採購單清單，用戶從清單中選擇要收貨的採購單。

**關鍵輸出**:
- `headerList`: 採購單清單陣列
- `wmsAsnNumber`: WMS 單號主鍵值 (後續 API 必用)
- `externReceiptNumber`: 採購單號
- `warehouseCode`: 實體倉庫代碼

**詳情請參閱**: [R1-取得預期收貨清單.md](R1-取得預期收貨清單.md)

### R2 - 取得收貨明細資料

依採購單號取得收貨明細，包含 PO 項次、貨號、品名、到期日、件數、漁廠名稱等資訊。

**關鍵輸出**:
- `itemList`: 明細資料陣列
- `lineNo`: WMS 項次 (R3 必用)
- `sku`: 貨號
- `descr`: 品名
- `expiryDate`: 到期日
- `packQty`: 件數
- `fishingGroundName`: 漁廠名稱

**詳情請參閱**: [R2-取得收貨明細資料.md](R2-取得收貨明細資料.md)

### R3 - 回傳收貨資料

輸入收貨資料並暫存至 WMS，同一 `lineNo` 可多次更新。

**關鍵輸入**:
- `requestFnName`: 區分 `"3.2確認收貨"` (新增) 或 `"3.3修改"` (更新)
- `lineNo`: WMS 項次
- `packQty`: 件數 (必填，需為整數)
- `expiryDate`: 到期日

**條件式必填欄位** (依 R2 的 `itemColumnConfig` 決定):
- `qty`: 數量 (小數)
- `batchNumber`: 生產批號
- `mfgDate`: 製造日
- `storageStatus`: 倉別
- `stockType`: 區位

**詳情請參閱**: [R3-回傳收貨資料.md](R3-回傳收貨資料.md)

### R4 - 取得收貨核對明細

完成所有收貨項目後，取得核對資料與儲位選單。

**關鍵輸出**:
- `verifyList`: 核對明細陣列
- `poTotalQty`: PO 總需求公斤數
- `recTotalPackQty`: 已收總件數
- `locList`: 儲位選單陣列
- `locCode`: 儲位編號 (R5 必用)

**業務邏輯**:
- 計算每個 PO 項次的收貨完成度
- 提供下拉儲位選單供用戶選擇

**詳情請參閱**: [R4-取得收貨核對明細.md](R4-取得收貨核對明細.md)

### R5 - 回傳全收確認

輸入板數與儲位，確認過帳並產生板標籤。

**關鍵輸入**:
- `recPalletQty`: 總板數 (整數)
- `recLocCode`: 儲位編號

**關鍵輸出**:
- `palletLabelList`: 板標籤清單
- `成功訊息`: 包含板數、儲位等資訊

**檢核規則**:
- 檢查所有 PO 項次是否皆有件數
- 確認單據狀態為「收貨中」才可執行

**詳情請參閱**: [R5-回傳全收確認.md](R5-回傳全收確認.md)

## 使用範例流程

### Android App 實作流程

```kotlin
// 1. 登入取得 Token (A1)
// 構造內層請求數據
val loginRequest = mapOf(
    "requestId" to UUID.randomUUID().toString(),
    "account" to account,
    "password" to password
)

// 加密並構造外層請求
val encryptedRequest = AesUtil.encryptRequest(loginRequest, aesKey, aesIv)

// 發送請求並解密回應
val token = AuthRepository.login(encryptedRequest)

// 2. 取得預期收貨清單 (R1)
val r1Data = mapOf(
    "requestId" to UUID.randomUUID().toString(),
    "storerCode" to "97286918",
    "docStatus" to "OPEN"
)
// 加密並發送請求，解密回應
val headerList = ReceivingRepository.getReceivingHeaders(r1Data)

// 3. 用戶選擇採購單
val selectedHeader = headerList[0]
val wmsAsnNumber = selectedHeader.wmsAsnNumber

// 4. 取得收貨明細 (R2)
val details = ReceivingRepository.getReceivingDetails(wmsAsnNumber, storerCode, externReceiptNumber)

// 5. 逐項輸入收貨資料 (R3)
details.forEach { detail ->
    // 用戶輸入收貨資料
    val packQty = getUserInputPackQty(detail)
    val expiryDate = getUserInputExpiryDate(detail)

    // 確認收貨 (requestFnName: "3.2確認收貨")
    ReceivingRepository.saveReceivingItem(
        wmsAsnNumber = wmsAsnNumber,
        storerCode = storerCode,
        externReceiptNumber = externReceiptNumber,
        lineNo = detail.lineNo,
        packQty = packQty,
        expiryDate = expiryDate,
        requestFnName = "3.2確認收貨"
    )
}

// 6. 用戶修改某項收貨資料
val itemToModify = details[0]
val newPackQty = 100

// 修改收貨 (requestFnName: "3.3修改")
ReceivingRepository.saveReceivingItem(
    wmsAsnNumber = wmsAsnNumber,
    storerCode = storerCode,
    externReceiptNumber = externReceiptNumber,
    lineNo = itemToModify.lineNo,
    packQty = newPackQty,
    expiryDate = itemToModify.expiryDate,
    requestFnName = "3.3修改"
)

// 7. 完成收貨，取得核對明細 (R4)
val verifyData = ReceivingRepository.getReceivingVerifyData(wmsAsnNumber, storerCode, externReceiptNumber)

// 8. 用戶選擇儲位
val selectedLoc = verifyData.locList[0]

// 9. 輸入板數
val recPalletQty = getUserInputPalletQty()

// 10. 確認全收 (R5)
val result = ReceivingRepository.confirmReceipt(
    wmsAsnNumber = wmsAsnNumber,
    storerCode = storerCode,
    externReceiptNumber = externReceiptNumber,
    recPalletQty = recPalletQty,
    recLocCode = selectedLoc.locCode
)

// 11. 列印板標籤
printPalletLabels(result.palletLabelList)
```

## 錯誤處理常見情境

### 情境 1: Token 失效

錯誤代碼：F119 (HTTP 401)

處理方式：
```kotlin
if (error.code == "F119") {
    // 清除舊 Token
    clearAccessToken()
    // 重新登入
    AuthRepository.login(account, password)
    // 重試原請求
    retryRequest()
}
```

### 情境 2: 採購單已過帳

錯誤代碼：F983 (HTTP 400)

處理方式：
```kotlin
if (error.message.contains("已過帳") || error.message.contains("已入庫")) {
    // 提示用戶採購單已完成
    showWarning("此採購單已完成收貨")
}
```

### 情境 3: R5 檢核失敗 - 項次未收完

錯誤代碼：F983 (HTTP 400)

處理方式：
```kotlin
if (error.message.contains("未有件數")) {
    // 顯示未收項目提示
    showWarning("所有 PO 項次必須都有件數才能全收確認")
    // 引導用戶返回步驟 5 補充收貨
    navigateBackToStep5()
}
```

## 注意事項

1. **RequestId 管理**
   - 每個請求必須產生唯一的 GUID
   - 同一操作的重試請求應使用相同的 requestId

2. **資料暫存**
   - R3 的收貨資料是暫存，未過帳前可修改
   - R5 確认後才正式過帳

3. **逾時處理**
   - 建議設定合理的請求逾時時間（如 30 秒）
   - 網路不稳時適當重試

4. **並發控制**
   - 同一採購單同一時間只能有一人操作
   - F984 錯誤表示請求重複

5. **狀態管理**
   - 用戶端應記錄當前操作採購單的狀態
   - 避免重複呼叫 R5 確認
